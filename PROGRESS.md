# proev.ru — журнал проекта

Живой документ: что сделано, как устроено, какие решения приняты и почему,
что осталось. Обновляется по ходу разработки.

## Что такое proev.ru

Сайт (а дальше — и приложение) для владельцев электромобилей в России:
карта зарядных станций с живыми статусами от водителей, каталог сервисов
(СТО, установка домашних зарядок, страхование) с лидогенерацией как
основной монетизацией, блог, админка для модерации.

## Архитектура

```
                    ┌─────────────────────┐
  Интернет ────────▶│  Nginx Proxy Manager │  (отдельная VM, приватная сеть)
                    │  TLS, домен, роутинг │
                    └──────────┬───────────┘
                               │ приватная сеть
                               ▼
                    ┌─────────────────────┐
                    │   VPS proev backend  │
                    │  ┌────────────────┐  │
                    │  │ NestJS backend │  │  порт 3001, открыт ТОЛЬКО
                    │  │ + AdminJS /admin│  │  на приватном интерфейсе
                    │  └───────┬────────┘  │
                    │          │           │
                    │  ┌───────▼────────┐  │
                    │  │  PostgreSQL    │  │  (внутри docker-сети,
                    │  │  + PostGIS     │  │   не проброшен наружу)
                    │  └────────────────┘  │
                    └─────────────────────┘
```

Фронтенд (Next.js) пока не задеплоен — существует как код, готовый к
деплою на любой Node-хостинг (Vercel, тот же VPS, etc.), обращается к
бэкенду через `NEXT_PUBLIC_API_URL`.

## Технологии

- **Backend:** NestJS + TypeScript, Prisma ORM, PostgreSQL 16 + PostGIS
- **Frontend:** Next.js (App Router) + Tailwind + MapLibre GL
- **Админка:** AdminJS (смонтирована вручную в `main.ts`, не как Nest-модуль — см. ниже почему)
- **Инфраструктура:** Docker Compose на VPS, reverse-proxy — Nginx Proxy Manager на отдельной VM

## Что реализовано

### Backend (NestJS + Prisma)
- Модели: `ChargingStation`, `StationReview`, `ServiceCategory`, `ServiceProvider`, `Lead`, `Article`, `User`, `Integration`
- REST API: станции (список, поиск рядом по радиусу, отзывы), каталог сервисов, лиды, статьи
- Роли пользователей: `user`, `partner`, `moderator`, `admin`

### Админка (`/admin`)
- AdminJS с ролевым разграничением: `admin` — полный доступ, `moderator` — только модерация станций/отзывов и лиды, без удаления и без доступа к пользователям/интеграциям
- Создание первого админа/модератора — только через CLI-скрипт (`npm run create-admin`), регистрации через UI нет и не будет

### Движок интеграций
- Таблица `Integration` — универсальное хранилище API-ключей внешних сервисов (сейчас: OpenChargeMap, заготовки под Яндекс.Карты и 2GIS), управляется прямо из `/admin -> Интеграции`
- Правки применяются без передеплоя (кэш на бэкенде — 1 минута)
- `.env`-переменная `OCM_API_KEY` остаётся как запасной вариант для локальной разработки

### Наполнение карты станциями
- `npm run seed` — импорт из OpenChargeMap (постранично, с ретраями и таймаутом — соединение с VPS до внешних API оказалось не всегда стабильным) + поддержка ручного списка проверенных станций (`manual-stations.json`)

### Дизайн
- Фирменный стиль: тёмно-синий (`ink`) + электрик-циан (`volt`) + янтарный (`signal`) акценты, шрифты Golos Text + JetBrains Mono
- Фирменный визуальный приём — «маршрутная лента» (линия-дорога с узлами станций), используется в хиро и как разделитель секций — отсылка к реальной проблеме больших расстояний в России
- Реализовано в коде: `frontend/src/components/RouteStrip.tsx`, применено на главной странице

### Инфраструктура
- `infra/docker-compose.yml` — Postgres+PostGIS и бэкенд, приватная docker-сеть
- `infra/deploy.sh` — единый скрипт разворачивания: Docker, firewall, `.env` с секретами, сборка, seed, создание админа
- TLS и домен — на отдельной VM с Nginx Proxy Manager (см. ниже, почему не Caddy)

## Ключевые решения и почему

| Решение | Почему |
|---|---|
| `bcryptjs` вместо `bcrypt` | `bcrypt` — нативный модуль, требует компиляции (Python, build tools) или скачивания прекомпилированного бинарника с GitHub, что оказалось нестабильным с этого VPS (ETIMEDOUT). `bcryptjs` — чистый JS, работает везде одинаково |
| `node:20-slim` вместо `node:20-alpine` | Prisma-движку нужен OpenSSL определённой версии; на Alpine/musl автоопределение регулярно ломалось. Debian-based slim — предсказуемее |
| AdminJS монтируется в `main.ts` через `dynamicImport()`, а не как Nest-модуль | Весь пакетный набор AdminJS v7 — чистый ESM (`"type": "module"`), а `@Module()`-декораторы Nest вычисляются синхронно при загрузке файла, что несовместимо с `require()` ESM-пакетов. `new Function('specifier', 'return import(specifier)')` — обходной приём, скрывающий вызов от даунлевела TypeScript в `require()` |
| `prisma db push` вместо `prisma migrate deploy` | В проекте пока нет файлов миграций (не было живой БД на момент написания схемы). `db push` синхронизирует схему напрямую. Стоит перейти на нормальные миграции, когда схема стабилизируется |
| Nginx Proxy Manager вместо Caddy | На сервере уже развёрнут NPM на отдельной VM как единая точка входа для всей приватной сети. Caddy на этом VPS просто конкурировал за порты 80/443 и не мог получить сертификат (ACME-challenge не долетал снаружи) |
| Ключи внешних API — в БД (`Integration`), а не только в `.env` | Правки без передеплоя, доступно из админки владельцу без доступа к серверу |

## Известные грабли (на будущее)

- **Windows + git push**: свежая распакованная папка не помнит `git remote` — нужно `git remote add origin ...` заново при каждой чистой распаковке архива
- **CRLF от Windows-терминала**: интерактивный `read` в bash-скриптах может получить `\r` в конце ввода от некоторых Windows SSH-клиентов — `deploy.sh` уже чистит это (`${VAR%$'\r'}`)
- **OpenChargeMap требует API-ключ** и отдаёт нестабильные большие ответы — сид разбит на страницы по 250 с ретраями
- **git conflicts на сервере**: если вносить ручные правки прямо на VPS для отладки (а мы это делали много раз в процессе), они конфликтуют с `git pull` из репозитория — тогда нужно `git reset --hard HEAD && git pull`, приняв версию из репозитория как источник истины
